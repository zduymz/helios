## Broke
- First `docker cp` of the GGUF into the kind node failed silently because `/models` didn't exist on the node. `hostPath` with `type: Directory` requires the dir to pre-exist — fix: `docker exec helios-control-plane mkdir -p /models` first.
- KServe `quick_install.sh` failed on macOS bash 3.2 with `readarray: command not found` (bash 4+ builtin). Worked around by manually `helm upgrade -i kserve-resources` with the same flags the script would've used.
- Re-running `quick_install.sh` after the partial failure errored on Istio (`cannot re-use a name that is still in use`) — the script uses `helm install` not `helm upgrade -i` for Istio, so it isn't idempotent.
- First `kubectl apply` of the `InferenceService` was rejected by Knative's webhook: `HostPath volume support is disabled`. Knative gates pod-spec fields behind feature flags; fix is to patch the `KnativeServing` CR (the operator reverts plain ConfigMap edits) with `spec.config.features."kubernetes.podspec-volumes-hostpath": "enabled"`.

## Learned
- "Deploy → Ready" baseline (warm image, warm model on disk): **~12 s**. That's the number to beat — or at least to *understand* — when KServe + Knative replaces this in 1c with scale-from-zero.
- `hostPath` is the cheapest dev shortcut and the single most node-coupled choice you can make. In prod it's PVC + RWX or an init-container that pulls from S3. KServe's `storageUri` is the second pattern wrapped in a CRD.
- The OpenAI API surface from 1a survived the move into Kubernetes unchanged. That's the point.
- **Cold start can be faster than the readiness probe's `initialDelaySeconds`** because Knative's queue-proxy starts routing on TCP-open, not on kubelet `Ready`. Saw the predictor pod at `1/2 Ready` while the curl was returning `200`. The kubelet readiness column underreports actual serving status.
- **Panic mode is real.** `scaleTarget: 1` + a single cold request → 2 pods spun up, because Knative's burst-concurrency calculation flipped to panic during the activator hand-off. Tune `scaleTarget` higher if pod-count surprises matter.
- The activator/autoscaler/queue-proxy split is *the* mental model for Knative: activator buffers when zero pods, autoscaler decides count from queue-proxy's concurrency reports, queue-proxy is the per-pod traffic gate. Once that clicks, every Knative log/metric makes sense.
- Knative + Istio + cert-manager + KServe is ~4 control planes' worth of operational surface. The trade is that Backstage golden paths in Phase 8 will template against *one* CRD instead of an HPA + Service + Gateway tuple. ADR: [`decisions/0001-kserve-over-raw-deployment.md`](../decisions/0001-kserve-over-raw-deployment.md).
